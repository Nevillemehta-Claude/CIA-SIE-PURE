CORE KERNEL

IMMUTABLE The foundational principles that govern all development activities. These cannot be modified or overridden.

First Principles

AXIOMS Immutable truths of software development.

The 8 Gold Standard Principles
1
NASA-Style Rigour

Apply aerospace-grade discipline to software development. Every decision documented, every assumption validated, every outcome verified.

2
Audit Before Build

Understand the current state completely before making changes. No assumptions without evidence.

3
Zero Drift Policy

Documentation and implementation must remain synchronized. Drift is debt; debt compounds.

4
Living Documentation

Documentation is not an afterthought. It evolves with the code, tested alongside the code.

5
Full Bidirectional Traceability

Every requirement traces to implementation. Every implementation traces to requirement. No orphans.

6
Evidence-Based Validation

Assertions require proof. Tests are evidence. Passing tests are the only acceptable evidence.

7
Single Source of Truth (SSOT)

One authoritative source per artifact. Duplication is divergence waiting to happen.

8
Defensive Documentation

Document for the skeptic. Assume the reader questions everything. Provide evidence for every claim.

The 5 Immutable Laws
#LawMeaning
1Meaning Precedes ImplementationUnderstand what before how. Define semantics before syntax.
2Structure Precedes IntelligenceGet the architecture right before adding complexity.
3Validation Precedes OptimizationMake it work, make it right, then make it fast.
4Explicit Precedes ImplicitNo magic. No hidden behavior. Everything visible and documented.
5Reversibility Precedes CommitmentEvery decision should be reversible until proven correct.
The 6 Architecture Principles
  1. Single Source of Truth (SSOT) - One authoritative source per data element
  2. Loose Coupling - Components interact through well-defined interfaces
  3. High Cohesion - Related functionality grouped together
  4. Explicit Contracts - All interfaces formally specified
  5. Fail-Safe Defaults - System fails to safe state
  6. Deterministic Behavior - Same input always produces same output

Modularity Doctrine

ARCHITECTURE The architecture of autonomous units.

The 7 Properties of a True Module
PropertyDefinitionTest
Self-ContainedAll dependencies explicit and internalCan be understood without external context
Independently TestableCan be tested in isolationHas its own test suite that runs independently
VersionedHas explicit version numberVersion changes when interface changes
DocumentedPurpose, interface, and behavior documentedNew developer can use without asking questions
Interface-DefinedInteraction only through defined interfaceNo direct access to internals
ReplaceableCan be swapped for alternative implementationInterface remains stable across implementations
EvolvableCan be updated without affecting dependentsInternal changes don't break consumers
Anti-Patterns to Avoid
  • Hidden Dependencies - Undeclared requirements on external components
  • Circular Dependencies - A depends on B depends on A
  • God Module - One module that does everything
  • Leaky Abstraction - Internal details exposed through interface
  • Version Lock - Rigid version requirements preventing updates

Verification Doctrine

TESTING The architecture of certainty. Testing is not a phase, it is a constant companion.

The 10 Verification Layers
10
ACCEPTANCE - Does it meet business goals?
9
SECURITY - Are there vulnerabilities?
8
PERFORMANCE - Is it within acceptable bounds?
7
END-TO-END - Do complete workflows work?
6
FUNCTIONAL - Do features work correctly?
5
REGRESSION - Did we break anything?
4
INTEGRATION - Do components connect properly?
3
CONTRACT - Are API contracts honored?
2
UNIT - Do individual functions work?
1
SYNTAX - Does the code compile/run?

RULE: Lower layers must pass before higher layers are tested.
