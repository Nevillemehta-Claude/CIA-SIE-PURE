# PHASE 01: IDEATION
## Standards

**Purpose:** Quality standards for idea capture and initial assessment

---

## IDEA CAPTURE STANDARDS

### Required Information

Every idea submission MUST include:

| Field | Required | Description |
|-------|----------|-------------|
| Title | ✓ | Clear, descriptive name (5-10 words) |
| Submitter | ✓ | Who proposed this idea |
| Date | ✓ | When submitted |
| Problem Statement | ✓ | What problem does this solve |
| Proposed Solution | ✓ | High-level solution concept |
| Target Users | ✓ | Who benefits from this |
| Expected Value | ✓ | Why this matters |

### Optional Information

| Field | Description |
|-------|-------------|
| Rough Effort | T-shirt size (S/M/L/XL) |
| Dependencies | Known blockers or prerequisites |
| Alternatives | Other ways to solve the problem |
| Risks | Known concerns |
| References | Related materials, prior art |

---

## IDEA DOCUMENTATION FORMAT

### Standard Idea Template

```markdown
# Idea: [Title]

**ID:** IDEA-[YYYY]-[NNN]
**Submitter:** [Name]
**Date:** [Date]
**Status:** Submitted | Under Review | Approved | Deferred | Rejected

## Problem Statement
[What problem exists? Who is affected? What is the impact?]

## Proposed Solution
[High-level description of the solution concept]

## Target Users
[Who will benefit from this solution?]

## Expected Value
[Why is this worth pursuing? What outcomes expected?]

## Initial Assessment
- Rough Effort: [S/M/L/XL]
- Strategic Alignment: [High/Medium/Low]
- Technical Feasibility: [High/Medium/Low]
- Resource Availability: [Yes/No/Partial]

## Notes
[Any additional context, references, or considerations]
```

---

## IDEA NAMING STANDARDS

### ID Format

`IDEA-[YYYY]-[NNN]`

- YYYY: Year submitted
- NNN: Sequential number (001-999)

Example: `IDEA-2025-042`

### Title Guidelines

| Do | Don't |
|----|-------|
| Be specific | Be vague |
| Use action words | Use passive voice |
| Keep under 10 words | Write sentences |
| Focus on outcome | Focus on technology |

**Good:** "Automated Invoice Processing for AP Team"
**Bad:** "New System Enhancement Project"

## ASSESSMENT STANDARDS

### Strategic Alignment Criteria

| Rating | Criteria |
|--------|----------|
| High | Directly supports strategic objectives |
| Medium | Indirectly supports or enables strategy |
| Low | Tangential to current strategy |
| None | Does not align with strategy |

### Technical Feasibility Criteria

| Rating | Criteria |
|--------|----------|
| High | Known technology, proven approach |
| Medium | Some unknowns, manageable risk |
| Low | Significant unknowns, high risk |
| Unknown | Requires research to assess |

### Effort Estimation (T-Shirt Sizing)

| Size | Duration | Team Size | Description |
|------|----------|-----------|-------------|
| S | < 2 weeks | 1-2 people | Simple, well-understood |
| M | 2-6 weeks | 2-4 people | Moderate complexity |
| L | 6-12 weeks | 4-8 people | Complex, multiple components |
| XL | > 12 weeks | 8+ people | Large initiative, multiple teams |

## IDEA REVIEW STANDARDS

### Review Cadence

| Volume | Frequency |
|--------|-----------|
| < 5 ideas/month | Ad-hoc |
| 5-20 ideas/month | Weekly |
| > 20 ideas/month | Twice weekly |

### Review Participants

| Role | Responsibility |
|------|----------------|
| Facilitator | Run meeting, ensure process |
| Business Rep | Assess value and priority |
| Technical Rep | Assess feasibility |
| Submitter | Clarify and defend idea |

### Review Outcomes

| Outcome | Next Step |
|---------|-----------|
| Approved | Move to Phase 2 (Concept) |
| Deferred | Add to backlog with rationale |
| Rejected | Document reason, notify submitter |
| Needs Info | Return to submitter with questions |

## PRIORITIZATION STANDARDS

### Prioritization Criteria

| Criterion | Weight | Description |
|-----------|--------|-------------|
| Strategic Value | 30% | Alignment with objectives |
| Business Impact | 25% | Revenue, cost, risk reduction |
| Effort Required | 20% | Resources and time needed |
| Technical Risk | 15% | Complexity and unknowns |
| Dependencies | 10% | Blockers and prerequisites |

### Priority Scoring

| Score | Meaning |
|-------|---------|
| 5 | Critical - Must do |
| 4 | High - Should do soon |
| 3 | Medium - Important but not urgent |
| 2 | Low - Nice to have |
| 1 | Minimal - Consider deferring |

## COMMUNICATION STANDARDS

### Submitter Communication

| Event | Timeline | Method |
|-------|----------|--------|
| Acknowledgment | 24 hours | Email/ticket |
| Review scheduled | 1 week | Email |
| Decision made | Same day as review | Email |
| Rejection reason | With decision | Email |

### Status Updates

Ideas in review should update status:
- Weekly if actively being assessed
- Upon any status change
- When additional information received

---

## DOCUMENTATION STANDARDS

### Idea Repository

All ideas MUST be stored in a central repository with:
- Searchable index
- Status tracking
- History preservation
- Access for all stakeholders

### Retention

| Status | Retention |
|--------|-----------|
| Approved | Permanent (moves to project) |
| Deferred | 2 years minimum |
| Rejected | 1 year minimum |

### Audit Trail

Every status change MUST record:
- Who made the change
- When it occurred
- Why it changed
- Previous status
